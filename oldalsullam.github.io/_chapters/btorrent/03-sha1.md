## Why SHA-1?

BitTorrent uses SHA-1 for two things:

1. **Info hash** — uniquely identifies a torrent across all peers and trackers
2. **Piece verification** — each downloaded piece is hashed and compared against the stored hash

SHA-1 produces a 160-bit (20-byte) output from any input. Same input always produces same output.

## Algorithm Overview

SHA-1 processes data in **64-byte blocks**:

```
Input → Pad to multiple of 512 bits
      → Split into 512-bit blocks
      → For each block:
            Expand 16 words → 80 words (message schedule)
            Run 80 rounds of mixing on 5 state variables
            Add results back to state
      → Final state = 20-byte hash
```

## The 5 State Variables

SHA-1 maintains 5 × 32-bit words, initialized to magic constants:

```c
uint32_t h[5] = {
    0x67452301,  // fractional part of sqrt(2)
    0xEFCDAB89,  // fractional part of sqrt(3)
    0x98BADCFE,  // fractional part of sqrt(5)
    0x10325476,  // fractional part of sqrt(7)
    0xC3D2E1F0   // fractional part of sqrt(11)
};
```

"Nothing up my sleeve" numbers — chosen to prove the constants aren't backdoored.

## Padding Rules

Before processing, the message is padded to a multiple of 512 bits:

1. Append `0x80` (the single '1' bit)
2. Append `0x00` bytes until length ≡ 448 (mod 512)
3. Append the original message bit-length as 64-bit big-endian

## Message Schedule Expansion

Each 64-byte block provides 16 × 32-bit words. We expand to 80 words:

```c
for (int i = 16; i < 80; i++) {
    W[i] = ROTL(W[i-3] ^ W[i-8] ^ W[i-14] ^ W[i-16], 1);
}
```

## The 80 Rounds

Four groups of 20 rounds, each with a different boolean function and constant:

| Rounds | Function | Constant K |
|--------|----------|------------|
| 0–19   | `Ch(b,c,d) = (b&c)|(~b&d)` | `0x5A827999` |
| 20–39  | `Par(b,c,d) = b^c^d` | `0x6ED9EBA1` |
| 40–59  | `Maj(b,c,d) = (b&c)|(b&d)|(c&d)` | `0x8F1BBCDC` |
| 60–79  | `Par(b,c,d) = b^c^d` | `0xCA62C1D6` |

Each round:

```c
uint32_t temp = ROTL(a, 5) + f(b,c,d) + e + K + W[i];
e = d; d = c; c = ROTL(b, 30); b = a; a = temp;
```

## Big-Endian Byte Order

SHA-1 is big-endian. On x86 (little-endian), you must byte-swap when loading words:

```c
uint32_t w = ((uint32_t)buf[0] << 24) |
             ((uint32_t)buf[1] << 16) |
             ((uint32_t)buf[2] <<  8) |
             ((uint32_t)buf[3]);
```

## Using SHA-1 in btorrent

```c
// One-shot
uint8_t hash[20];
sha1(data, len, hash);

// Streaming (for large files)
SHA1_CTX ctx;
sha1_init(&ctx);
sha1_update(&ctx, chunk1, len1);
sha1_update(&ctx, chunk2, len2);
sha1_final(&ctx, hash);
```

## Verifying a Piece

```c
uint8_t computed[20];
sha1(piece_data, piece_length, computed);

const uint8_t *expected = torrent->pieces_hash + (i * 20);

if (memcmp(computed, expected, 20) == 0) {
    // Piece is authentic — write to disk
} else {
    // Corrupted — discard and re-request from another peer
}
```

<div class="callout-title">SHA-1 Is Broken — But That's OK Here</div>
<p>SHA-1 collision attacks exist (SHAttered, 2017). But BitTorrent uses SHA-1 for integrity verification of <em>known content</em>, not authentication of <em>unknown signers</em>. A peer sending you corrupted data would have to find a collision that still produces the correct hash — much harder than just sending correct data.</p>
